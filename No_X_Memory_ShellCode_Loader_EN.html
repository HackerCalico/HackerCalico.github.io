No_X_Memory_ShellCode_Loader

1. Introduction

It's a no-kill project and has nothing to do with PWN!

A new ShellCode loader that loads ShellCode without executable permissions.

Project: https://github.com/HackerCalico/No_X_Memory_ShellCode_Loader

Blog: https://hackercalico.github.io/No_X_Memory_ShellCode_Loader.html

The technology was first proposed in early 2024 when a very simple x86 sample was released to prove feasibility. Because of the tight schedule this year, I haven't released the x64 version until now, which can be said to be a fairly complete version.

I'm thrilled with this breakthrough!

2. Circumvent advantages

No decryption, no X memory, directly load the ShellCode ciphertext in the running R memory.

The following features are avoided:

(1) Request memory for RWX attributes.

(2) Modify the memory properties of W and X back and forth.

(3) A ShellCode feature appears in memory.

3. Technical Principles

Loading process:

(1) Generate a ShellCode machine code normally.

(2) ShellCode --- Converter ---> Custom assembly directives.

(3) The interpreter runs the custom assembly instruction.

Details:

https://hackercalico.github.io/No_X_Memory_ShellCode_Loader_Principle.html

(3) Generate custom assembly instructions

Converts the original assembly directive in the asm.txt to a custom assembly directive.

> python Converter.py
1.反汇编
2.Imul转换
3.生成自定义汇编指令
选择: 3
0_4_q_pq70+i20_q_q38_......2ed_3_q__q__!

PVOID mnemonicMapping[] = { Push, Pop, ......, Jle };

0_4_q_pq70+i20_q_q38 for the first custom assembly directive,! Customize the end flag for the entire assembly directive.

The original assembly directive for the first article: 0x00 mov qword ptr [rsp + 0x20], r9

Command address: 0x00 ------> 0

It needs to be used when dealing with JCC jump instructions, removing 0x to shorten the length.

Mnemonic: MOV ------> 4

4 is the subscript of mov in mnemonicMap.

The interpreter executes instructions one by one, and obtains the handler pointer of the current instruction in mnemonicMap through subscripts to make reflection calls. Avoided a lot of if else or switch cases in the interpreter code.

Operand 1: Qword PTR [RSP + 0x20] ------> q_pq70 + I20

q stands for QWORD and p stands for ptr.

q70 represents vtRSP. The value of the virtual register is stored in the interpreter via an array of vtRegs, 70 is the offset of the vtRSP relative to the base address of the vtRegs, and the value of the register is manipulated directly from the address. Eliminates the need for a large number of registers with different digits in the interpreter code, as well as tedious operations.

i20 means 0x20 immediately.

Op. 2: R9 ------> q_q38

q stands for QWORD.

q38 represents R9, the same is true.

6. Interpreter implementation

(1) Create virtual stacks and virtual registers

PVOID vtStack = malloc(0x10000);

DWORD_PTR vtRegs[18] = { 0 };
vtRegs[14] = vtRegs[15] = (DWORD_PTR)vtStack + 0x9000;

14 and 15 correspond to vtRSP and vtRBP, respectively.

(2) Set the initial value of the virtual register

Because the interpreter is simulated from the beginning of the ShellCode function, the parameters of the ShellCode function are constructed in the virtual space before the simulation begins.

The CMD command provided by this project executes the ShellCode function by calling the following code:

ExecuteCmd(commandPara, commandParaLength, &outputData, &outputDataLength, funcAddr);

The assembly directive corresponding to this line of code:

lea rax, [funcAddr]
mov qword ptr [rsp+20h], rax
lea r9, [outputDataLength]
lea r8, [outputData]
mov edx, dword ptr [commandParaLength]
lea rcx, [commandPara]
call ExecuteCmd

So you need to build the function parameters in the virtual space with the following code:

vtRegs[0] = (DWORD_PTR)pFuncAddr;
*(PDWORD_PTR)(vtRegs[14] + 0x20) = vtRegs[0];
vtRegs[7] = (DWORD_PTR)pOutputDataLength;
vtRegs[6] = (DWORD_PTR)pOutputData;
vtRegs[3] = commandParaLength;
vtRegs[2] = (DWORD_PTR)commandPara;
vtRegs[14] = vtRegs[14] - sizeof(DWORD_PTR);

(3) Parsing custom assembly instructions

The elements of each instruction are parsed according to the format of Instruction Address_Mnemonic_Bit 1_Omannamon 1_Number of Digits 2_ Operand 2.

The type and address of each operand can be obtained through the GetOpTypeAndAddr function, and the handler of each instruction will directly operate on its value through the address of the operand, which is very convenient.

If the operand is immediate, for example, i12 (0x12). The 12 strings are converted to numbers through the strtol function, and the address of the number is the address of the operand.

If the operand is the second operand or memory space of the LEA, for example, LQ70+i20 ([rsp + 0x20]) or pq70+i20 (PTR [rsp + 0x20]). The child element is parsed first, and the result is saved to number1. If the operand is the second operand of LEA, the address of number1 is the address of the operand. If the operand is memory space, the value of number1 is the address of the operand.

If the operand is a register, e.g. Q70 (RSP). 70 is the offset of vtRSP relative to the vtRegs base address, then vtRegs base address + 70 is the address of the operand.

(4) Call the handler of the corresponding instruction

The handler pointer of the current instruction is obtained by parsing the subscript of the current instruction.

PVOID mnemonicMapping[] = { Push, Pop, ......, Jle };
PVOID instructionFunc = mnemonicMapping[mnemonicIndex];

Here are some examples of handlers for several directives:

MOV directive

// 其他两个操作数的指令
else {
    ((void(*)(...))instructionFunc)(opType1, opBit1, opAddr1, opBit2, opAddr2);
}
void Mov(char opType1, char opBit1, DWORD_PTR opAddr1, char opBit2, DWORD_PTR opAddr2) {
    switch (opBit1)
    {
    case 'q':
        *(PDWORD64)opAddr1 = *(PDWORD64)opAddr2;
        break;
    case 'd':
        if (opType1 == 'r') {
            *(PDWORD_PTR)opAddr1 = *(PDWORD)opAddr2;
        }
        else {
            *(PDWORD)opAddr1 = *(PDWORD)opAddr2;
        }
        break;
    case 'w':
        *(PWORD)opAddr1 = *(PWORD)opAddr2;
        break;
    case 'b':
        *(PBYTE)opAddr1 = *(PBYTE)opAddr2;
        break;
    }
}

case 'd' indicates that operand 1 is DWORD, and there is a special case when operand 1 is of type DWORD register.

The special case is illustrated by the following example:

mov rax, 0x1234567812345678
mov eax, 0x11111111

After running rax is 0x0000000011111111.

JCC Directive

A handler pointer to a specific JCC instruction.

Jcc(instructionFunc, opAddr1, pVtRegs);

Use the handler of the specific Jcc instruction to determine whether to jump. If it jumps, the value of operand 1 is assigned to vtRIP.

void Jcc(PVOID instructionFunc, DWORD_PTR opAddr1, PDWORD_PTR pVtRegs) {
    DWORD_PTR vtEFL = pVtRegs[17];
    int isJmp = ((int(*)(...))instructionFunc)(vtEFL);
    if (isJmp) {
        DWORD_PTR vtRIP = *(PDWORD_PTR)opAddr1;
        pVtRegs[16] = vtRIP;
    }
}

JE Directive

As a handler for specific Jcc instructions, the value of vtEFL is assigned to the flag register first, and then it is determined whether to jump.

int Je(DWORD_PTR vtEFL) {
    int isJmp = 1;
    __asm {
        mov rax, vtEFL
        push rax
        popf
        je jmp
        mov isJmp, 0x00
        jmp :
    }
    return isJmp;
}

Call directive

The implementation is the most complex of all the instructions, as it involves calls to the Windows API.

First, save the real top and bottom of the stack, and finally restore the real top and bottom of the stack to ensure that the interpreter can run normally.

Before calling the Windows API, you need to overwrite the value of the virtual register with the value of the real register, which is equivalent to constructing the parameters of the Windows API.

Once you've called the Windows API, overwrite the value of the virtual register with the value of the real register.

void Call(DWORD_PTR opAddr1, PDWORD_PTR pVtRegs) {
    // 保存真实栈顶栈底
    DWORD_PTR realRSP;
    DWORD_PTR realRBP;
    __asm {
        mov realRSP, rsp
        mov realRBP, rbp
    }

    // Window API 地址
    DWORD_PTR winApiAddr = *(PDWORD_PTR)opAddr1;

    // 虚拟寄存器 覆盖 真实寄存器
    DWORD_PTR vtRAX = pVtRegs[0];
    ......
    DWORD_PTR vtRBP = pVtRegs[15];
    __asm {
        mov rax, vtRAX
        ......
        mov rsp, vtRSP
        // mov rbp, vtRBP (与 Call 冲突)
    }

    // 调用 Windows API
    __asm {
        call qword ptr[winApiAddr] // (call qword ptr [rbp])
    }

    // 保存调用 Windows API 后真实寄存器的值
    __asm {
        push rax
        ......
        push rbp
    }

    // 真实寄存器 覆盖 虚拟寄存器
    DWORD_PTR currentRSP;
    __asm {
        mov currentRSP, rsp;
    }
    pVtRegs[0] = *(PDWORD_PTR)(currentRSP + 0x78); // RAX
    ......
    pVtRegs[14] = *(PDWORD_PTR)(currentRSP + 0x08) + 0x70; // RSP
    pVtRegs[15] = *(PDWORD_PTR)(currentRSP + 0x00); // RBP

    // 还原真实栈顶栈底
    __asm {
        mov rsp, realRSP
        mov rbp, realRBP
    }
}
