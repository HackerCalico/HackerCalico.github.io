Principle of Loading ShellCode without executable permission

1. Introduce

No need to decrypt, no X memory required, directly load the ShellCode ciphertext in the running R memory.

2. General ShellCode Loader

When everyone first starts learning ShellCode, they usually don’t understand what ShellCode itself is, but only learn how to write the following loader:

unsigned char buf[] = "ShellCode Ciphertext";

void* p = VirtualAlloc(NULL, sizeof buf, MEM_COMMIT, PAGE_EXECUTE_READWRITE);
memcpy(p, buf, sizeof buf);

// ShellCode Decryption

((void(*)())p)();

The above loader directly writes the ShellCode ciphertext into RWX (readable, writable and executable) memory for decryption and then calls it. At this time, a rare and sensitive RWX memory space appears in the process memory, which is easily detected and killed.

In order to avoid using the RWX memory attribute, everyone first writes the ShellCode ciphertext into the RW memory for decryption, and then changes the memory attribute to RX to run. If the backdoor program is directly generated by Hook CS, you will find that when executing some sensitive functions, the backdoor takes this operation of modifying memory attributes back and forth, which is easy to be detected and killed.

So I started thinking about whether there is a way to completely circumvent the above problems.

3. How ShellCode Works

In order to find a new way to load ShellCode, I decided to take a closer look at ShellCode.

ShellCode is a piece of address-independent machine code. Machine code is the hard-coding of the assembly instructions corresponding to the code. It usually exists in the .text section of the program file, such as the following MyMessageBoxA_Not function:

Hardcoded and assembly instructions for this function:

48 83 EC 38       ------> SUB RSP, 0X38
C6 44 24 20 00    ------> MOV BYTE PTR [RSP + 0X20], 0
41 B9 40 00 00 00 ------> MOV R9D, 0X40
4C 8D 44 24 20    ------> LEA R8, [RSP + 0X20]
48 8D 54 24 20    ------> LEA RDX, [RSP + 0X20]
33 C9             ------> XOR ECX, ECX
FF 15 2F 12 00 00 ------> CALL QWORD PTR [RIP + 0X122F]
48 83 C4 38       ------> ADD RSP, 0X38
C3                ------> RET

You can see that the Windows API MessageBoxA is called through the Call instruction, but it is obvious that the address of MessageBoxA is stored in other locations, so if you run this machine code alone, it will fail.

ShellCode address is irrelevant, which means that this external address is not used directly. The method of implementation is not to call the Windows API directly during the process of writing code, but to actively obtain the address of the Windows API and call it, such as the following MyMessageBoxA function:

typedef int(WINAPI* pMessageBoxA)(HWND, LPCSTR, LPCSTR, UINT);

#pragma code_seg(".shell")

void MyMessageBoxA(pMessageBoxA funcMessageBoxA) {
    char text[] = { '\0' };
    funcMessageBoxA(0, text, text, MB_ICONINFORMATION);
}

#pragma code_seg(".text")

int main() {
    MyMessageBoxA(MessageBoxA);
}

The address of MessageBoxA used by this function is passed in from the outside through parameters, so the machine code of this function can be run directly as ShellCode.

4. Implementation analysis of new loader

Through an in-depth understanding of ShellCode, we can know that ShellCode is actually the hard-coding of the assembly instructions corresponding to the code written according to the address-independent standard, and the assembly instructions correspond to the hard-coding.

So it can be said that running ShellCode is running its assembly instructions. As long as the equivalent function of its assembly instructions is realized, the equivalent operation of ShellCode is realized.

So the current research is transformed into what functions its assembly instructions implement.

By learning assembly language, you can know that these assembly instructions simply continuously modify the values ​​of registers, stacks, and memory, and construct the parameters required to call the Windows API through constant modification, and then successfully call the Windows API.

The construction process of function parameters can be simply explained through MyMessageBoxA above. The function is called through the following code:

MyMessageBoxA(MessageBoxA);

This line of code actually constructs the parameters of the function, and its assembly instructions are:

mov rcx,qword ptr [__imp_MessageBoxA]
call MyMessageBoxA

The assembly instructions put the address of MessageBoxA into the RCX register, which is a simple construction process. Complex processes, such as loop decryption of strings, etc., can be collectively considered as the process of constructor parameters.

So the current research is transformed into how to use other methods to construct Windows API parameters for calling.

The way I thought of was to implement an interpreter for assembly instructions. An interpreter is a program that performs lexical, grammatical, semantic, etc. analysis on code line by line.

As long as I pass in the text of the assembly instructions, the interpreter will parse the instructions one by one to implement the corresponding functions. Several issues are involved here. For example, when mov rsp, 0x00 is interpreted, the value of the real RSP register should not be changed to 0x00. This will cause an error in the interpreter itself. The solution is to implement virtual registers and virtual stacks, and change the virtual vtRSP to 0x00. When interpreting the calling instructions of the Windows API, first overwrite the real register with the value of the virtual register. At this time, the parameters of the Windows API are in a complete state, and then the Windows API can be called directly to succeed.

The following uses MyMessageBoxA as an example to demonstrate the explanation process:

Assembly instructions for this function:

MOV QWORD PTR [RSP + 8], RCX
SUB RSP, 0X38
MOV BYTE PTR [RSP + 0X20], 0
MOV R9D, 0X40
LEA R8, [RSP + 0X20]
LEA RDX, [RSP + 0X20]
XOR ECX, ECX
CALL QWORD PTR [RSP + 0X40]
ADD RSP, 0X38
RET

Simulation interpreter:

The following code ignores the parsing process of assembly instructions and directly simulates each instruction to modify the virtual value to construct the parameters of the Windows API. After overwriting the virtual value with the real value, the Windows API is successfully called.

// Virtual Stack
PVOID vtStack = malloc(0x10000);
// Virtual Stack Top
DWORD64 vtRSP = (DWORD64)vtStack + 0x9000;

// mov rcx,qword ptr [__imp_MessageBoxA]
DWORD64 vtRCX = (DWORD64)MessageBoxA;
// call MyMessageBoxA
vtRSP -= 8;

// MOV QWORD PTR [RSP + 8], RCX
*(PDWORD64)(vtRSP + 8) = vtRCX;
// SUB RSP, 0x38
vtRSP -= 0x38;
// MOV BYTE PTR [RSP + 0x20], 0
*(PBYTE)(vtRSP + 0x20) = 0;
// MOV R9D, 0x40
DWORD64 vtR9 = 0x40;
// LEA R8, [RSP + 0x20]
DWORD64 vtR8 = vtRSP + 0x20;
// LEA RDX, [RSP + 0x20]
DWORD64 vtRDX = vtRSP + 0x20;
// XOR ECX, ECX
vtRCX ^= vtRCX;

// Virtual Registers overwrite Real Registers
__asm {
    mov rcx, vtRCX
    mov rdx, vtRDX
    mov r8, vtR8
    mov r9, vtR9
    mov rsp, vtRSP
}

// CALL QWORD PTR [RSP + 0x40]
__asm {
    call qword ptr[rsp + 0x40]
}

// ADD RSP, 0x38
vtRSP += 0x38;

// RET
vtRSP += 8;
